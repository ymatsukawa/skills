開発事業部内 · CI/CD パイプライン再構築の投資提案

状況
  • Atlas の CI 実行時間が 1 年で 2.3 倍に肥大化
  • PR マージリードタイム中央値は 2.4 日
  • CI 月次費用は ¥2.4M に到達
問題点
  • E2E スイートの肥大化と flaky テストの放置
  • preview 環境が毎回フル再作成で重い
  • 開発者のフォーカス時間と CI への信頼が同時に劣化
将来の発生コスト
  • 年度末に +¥600K / 月のコスト超過の見込み
  • 部門 KPI(リードタイム / 障害率)双方の悪化
  • 並列化の延長線では解けない構造的な行き詰まり
解決道筋
  • エンジニア 2 名 × 4 か月で CI/CD を再構築
  • リードタイムを 0.6 日、CI 費用を ¥1.3M / 月 に圧縮
  • 投資回収は約 6 か月の見込み

事実の積み上げ

  • 01

    PR の マージリードタイム中央値は 2.4 日。1 年前(0.9 日)と比べて 2.7 倍に伸びている。

    github metrics 2026-Q1

    主因は CI 完走待ちと、レビュー後の main リベース・再 CI の繰り返し。 個別レビュアーの応答速度は同期間でほぼ横ばい。

  • 02

    CI ジョブの 約 62% の時間が E2E スイート 1 本(tests/e2e/full-suite)に集中している。

    CI レポート集計

    このスイートは 4 並列で実行されているが、テストデータのシードがシリアル依存で これ以上は分割できない。フィクスチャの整理から再設計が必要。

  • 03

    ステージング相当の preview 環境立ち上げに毎回 12〜18 分を要している。

    deploy log 2026-04

    Helm chart が単一の巨大 release になっており、変更の有無に関わらず全リソースを再作成している。 キャッシュやイメージ層の再利用が効いていない。

  • 04

    flaky テストによる CI 再実行が 週あたり平均 9 件 発生している。

    CI 再実行ログ

    原因の上位は外部 API のレートリミットと、テストごとの状態の取り違え。 一時的な隔離はしているものの、根本対処は後回しになっている。

  • 05

    CI ランナーの 月次費用は ¥2.4M。前年同月から +¥0.7M 増加している。

    クラウド請求

    実行時間の増加に比例しており、現状のままだと年度末には ¥3M / 月 規模に到達する見込み。
