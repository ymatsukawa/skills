作業依頼: A 社向け請求 CSV フォーマット変更の CS / Sales / dev 連携対応

やりたいこと

  • A 社が経理 SaaS への取り込みに使用している「月次請求 CSV」のフォーマットを、A 社の指定スキーマに合わせて切り替え可能にする
  • CS / Sales / dev の 3 部署で対応分担を明確化し、A 社への一次回答までを 5 営業日以内で完了させる
  • 対応合意までのコミュニケーションログを #cs-a-company Slack チャンネルに集約し、CS / Sales / dev 全員が同じ情報源を参照できる状態にする

なぜやるか

  • A 社は年額 ARR 上位 10 社に入る大口顧客で、本要望の遅延は次回更新時 (2026-09) の継続判断に影響する
  • 過去にも CS が拾った要望が Sales 経由で dev に渡らず、A 社側で 2 週間放置された事例があるため、今回は最初から 3 部署を巻き込んだフローで対応する
  • 同種の要望 (CSV カラム変更) は B 社・C 社からも上がっており、本対応をテンプレ化して横展開できる状態にしておきたい

実施事項: 概要

前提条件

  • A 社の契約上、フォーマット個別対応の SLA は定義されていない。今回はベストエフォートだが、5 営業日以内の一次回答を社内目標とする
  • 既存 CSV 出力は全顧客共通のフォーマットで稼働しており、A 社のためだけにグローバル変更は行わない
  • 本対応の起点は CS 経由の問い合わせ (Zendesk Ticket #4821)。元の要望文・添付サンプルは Zendesk 内が一次情報

確定

  • 対応方針は「テナント単位でフォーマット切替できる仕組みを dev が用意し、A 社のみ新フォーマットに切り替える」
  • CS の窓口は山田、Sales の主担当は鈴木、dev 側 PM は佐藤
  • A 社への一次回答期限は 2026-05-21 (木) 17:00
  • 対応に伴う追加費用は請求しない (関係性維持を優先)。営業上の判断は鈴木が決裁済み

未決

  • 新フォーマットに含める「税区分カラム」の値を、A 社の経理 SaaS が要求する 4 値で出すか、汎用の 2 値 + 注釈で出すか
    A 社経理担当へのヒアリング待ち。CS 山田から 2026-05-15 (水) 中に追加質問する予定。
  • 本機能を A 社の隣接顧客 (同業他社の B 社・C 社) に同条件で開放するか、A 社限定とするか
  • リリース予定日。dev チームの来週スプリント余力を見て確定 (5/20 のスプリントプランで判断)

実施事項: 詳細

手順

  1. CS 問い合わせ内容の整理と要件構造化 Zendesk Ticket #4821 を起票。A 社からの要望文・添付サンプル CSV を、社内ヒアリングシート (Notion: requests/2026-05-a-company) に転記し、「現状」「希望」「期限」の 3 項目に整理する。 期限: 2026-05-15 (水) 12:00 / 担当: 山田 (CS)
  2. CS Sales へのハンドオフ会 鈴木 (Sales) と 30 分の同期会を設定。要望の背景 (A 社の経理 SaaS 切替) と、契約面の影響範囲を引き継ぐ。会の結論を #cs-a-company に投稿する。 期限: 2026-05-15 (水) 17:00 / 担当: 山田 (CS) → 鈴木 (Sales)
  3. SALES 商務面の確認と A 社合意の取り付け A 社窓口 (経理部 田辺氏) と 1on1。追加費用なし・フォーマット切替方式での対応合意を口頭で取り、議事録を Salesforce 商談オブジェクトに保存。税区分カラムの仕様も同席で確認する。 期限: 2026-05-18 (月) 中 / 担当: 鈴木 (Sales)
  4. SALES dev チームへの要件提出 合意後に request-software-mvc.html 相当の依頼書テンプレートを使い、dev チームの起票キュー (Linear: BILL-*) にチケットを切る。「切替可能なフォーマット」「テナント単位での切替方法」の 2 点を明示する。 期限: 2026-05-19 (火) 10:00 / 担当: 鈴木 (Sales) → 佐藤 (dev PM)
  5. DEV 実装方針の見積もりと一次回答 佐藤 (dev PM) が起票チケットを受領し、設計案 (テナント設定キー billing.csv_format による切替) と工数見積を 24 時間以内に返す。設計案は Notion design/billing-csv-format-switch に置く。 期限: 2026-05-20 (水) 10:00 / 担当: 佐藤 (dev PM)
  6. CS A 社への一次回答 山田 (CS) が dev 側の設計案・想定リリース時期を取りまとめ、A 社へメール返信。Salesforce にも一次回答済みステータスを記録。 期限: 2026-05-21 (木) 17:00 / 担当: 山田 (CS)
  7. DEV 実装・QA・段階リリース 別途 dev チーム内のスプリントで実装。テナント切替フラグを A 社のみ ON にしてリリースし、初回出力後に CS 山田・Sales 鈴木が A 社の経理 SaaS への取り込み成功までを確認する。 期限: スプリント枠次第 (2026-05-29 を目標) / 担当: 佐藤 (dev PM) ・ 山田 (CS) ・ 鈴木 (Sales)

禁則事項

  • CS が単独で dev に直接ハンドオフしない。商務面の確認が抜けると追加費用の判断や契約解釈で齟齬が生じるため、必ず Sales を経由する
  • Sales が A 社にリリース日を確約しない。dev のスプリント枠が未確定の段階で日付を出すと、ずれた際の信頼毀損が大きい
  • dev が「A 社専用フォーマット」を直接ハードコードしない。あくまでテナント設定でスイッチする実装にし、横展開を可能にしておく
  • 本件のやりとりを DM や個人メールで完結させない。CS / Sales / dev のどの部署からも追えるよう #cs-a-company および Salesforce / Linear に必ず記録する
  • B 社・C 社へ「同じ要望なら対応します」と先回りで案内しない。横展開の方針が未決のため、判断確定までは A 社のみの個別対応として扱う
