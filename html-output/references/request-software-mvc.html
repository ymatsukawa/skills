作業依頼: ユーザー通知設定 (Notification Preferences) の追加

ゴール

  • ユーザーごとにメール / プッシュ / アプリ内通知の受信可否を設定できる画面を /settings/notifications に追加する
  • 通知の種別は marketing / product_update / security の 3 系統。security は法令上オフにできない仕様にする
  • 既存の UserMailer / PushDispatcher から、送信前に NotificationPreference を参照させる

なぜやるか

  • CS 経由で「マーケメールが多い、止めたい」という問い合わせが直近 1 ヶ月で 18 件発生しており、現状は CS が DB を直接更新して対応している
  • 2026 年 6 月施行の改正特定電子メール法対応として、ユーザー自身が受信設定を変更できる導線を提供する必要がある
  • 通知種別の細分化により、解約理由「通知が多すぎる」(直近四半期で 7.2%) を抑制する

現状

  • ユーザーが自分で通知受信可否を変更する導線が存在しない。停止希望はカスタマーサポート問い合わせを経由するしかなく、CS が DB を直接更新して凌いでいる (直近 1 ヶ月で 18 件)
  • users テーブルには通知関連カラムが一切なく、種別ごとの受信状態を保存する仕組みもない
  • 既存の UserMailer / PushDispatcher は opt-in 判定をせず、対象ユーザー全員に送信する実装になっている
  • 2026-06 施行の改正特定電子メール法に対し、ユーザー側で受信解除できる手段がない状態。法令対応が未着手
  • 解約アンケートで「通知が多すぎる」が直近四半期 7.2%。通知種別の粒度を切れないため、ピンポイントなオフが提供できていない

理想

  • /settings/notifications 画面でユーザー自身が marketing / product_update の受信可否を種別単位で切替できる
  • security 通知は法令上 OFF にできず、UI 上もチェックボックスが存在しない (固定文言のみ表示)。コントローラ / モデル両層で OFF を弾く二重防御
  • UserMailer / PushDispatcher が送信前に NotificationPreference を参照し、opt-out ユーザーには配信しない
  • 改正特定電子メール法 (2026-06 施行) の要件を満たす導線が提供されており、CS の手動 DB 更新が不要になっている
  • フィーチャーフラグ notification_preferences_v1 の段階公開 (5% → 25% → 100%) でリリースされ、配信実績の急減リスクが監視されている

決議事項

決定済

  • テーブル名は notification_preferences、user_id は一意制約 (1 ユーザー 1 レコード)
  • 通知種別は marketing / product_update / security の 3 値。スキーマは boolean × 3 カラム (jsonb は不採用)
  • security 通知は強制 ON。更新時にバリデーションで弾く
  • UI は既存 SettingsLayout 配下に新規ページを追加する

未決

  • 既存ユーザーへのデフォルト値を「全て ON」とするか「マーケのみ OFF」とするか
    マーケのみ OFF にすると配信数が初日から 30% 程度減るため、Growth チームとの調整が必要。来週月曜のレビューで方針決定予定。
  • プッシュ通知の OS 側設定 (iOS / Android のシステム設定) と本機能の関係をどう表記するか

禁則事項

  • 既存 users テーブルへのカラム追加で対応しない。今後の通知種別追加に備え、必ず別テーブルで管理する
  • カラム名に email_enabled など媒体名を含めない。現状はメール/プッシュを横串で扱うため、通知の「種別」で切る
  • security 通知を OFF にする抜け道を作らない。コントローラ側 strong params でも除外、モデル側 validate でも弾く二重防御
  • バックフィル時に通知メールを発火させない。insert_all を使い ActiveRecord コールバックを通さないこと
  • フィーチャーフラグ未通過の状態で routes を全ユーザーに公開しない。constraints でフラグ判定を挟む

備考

前提条件

  • 対象ブランチは main。リリースはフィーチャーフラグ notification_preferences_v1 配下で段階公開
  • Ruby 3.3 / Rails 7.1。devise による認証済みユーザー前提
  • 既存の users テーブルには通知関連カラムを持たない。新規テーブルとして切り出す

リスク管理

リスク 度合 原因 対策 担当
バックフィル中に users テーブルが長時間ロックされる High 本番ユーザー数 ≒ 280 万。単一トランザクションで insert_all を回すと所要 10 分超 disable_ddl_transaction! + 1,000 件バッチ。実行は深夜帯、進捗を JSONL で出力 dev チーム
opt-out 集中で marketing 系の配信実績が急減し売上に影響 Middle 初期デフォルト値の方針が未決。Growth との合意もまだ フィーチャーフラグで 5% → 25% → 100% の段階公開。各段でメール CTR を監視し閾値超過時は巻き戻し Growth 連携
既存の UserMailer のテストが opt-in 前提で書かれていて落ちる Middle marketing_digest のテスト fixture に notification_preference が無い fixture 一括更新 PR を先行マージ。CI で失敗パターンを事前に検出 dev チーム
